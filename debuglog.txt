[6/18/2026] [9:12 am] - Backend errors mistaken as frontend
  [Details]
    server.py: returning None from typed endpoints causes 500s
  [Cause]
        return None in Exception catching block violates promise to return MessageList (Pydantic),
        but returns None, causing FastAPI to raise a response validation error but the return response
        replaced with None, so it makes backend failures feel like frontend failures.
  [What I did]
        replaced return None with return MessageList[memo=[]] (con: makes errors look like they "work"
        by returning at the HTTP level without an assistant reply), raise an error at the Exception level instead
  [Solution]
        Do not return “empty valid data” for a failed operation.

        Think of backend responses as contracts:

        200 OK means “the operation succeeded”
        4xx means “the client sent something wrong”
        5xx means “the server failed while trying”
        an empty list should mean “successfully found nothing,” not “something crashed”
        For chat specifically:

        /chat success = user message was accepted, model replied, response contains messages
        /chat failure = no assistant reply should be trusted
